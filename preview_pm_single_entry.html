PM Dashboard - Single Entry Model

The PM enters only one number per GL: the 2027 Proposed Budget. The Increase $ and Increase % columns derive automatically from the difference vs. 2026 Current Budget. No formula juggling, no double-entry, no "wait, which column do I fill in?". Type the dollar amount you want. Done.

What changes for the PM: Today they decide whether to fill "Increase %" OR "Increase $" and the proposed budget is computed. New flow: type the proposed dollar amount directly, the deltas show as read-only badges next to it. Less thinking, fewer rounding bugs, faster to fill in.

Before (today)

PM types into Increase % or Increase $:

Management Fees5.0%$7,287$153,025

Three columns to think about. Proposed is auto-computed.

After (proposed)

PM types into 2027 Proposed Budget:

Management Fees$153,025+$7,287+5.0%

One number. Deltas live-update as you type.

Edge cases the new model handles cleanly: setting Proposed equal to Current shows 0%/$0 (the "no change" state). Setting Proposed to 0 shows a -100% delta. Negative current budgets (e.g. tax abatements as a contra) flip the sign correctly.

Implementation notes

  • Backend: proposed_budget becomes the source of truth for PM input. increase_pct and increase_dollar are derived on the fly from (proposed_budget - current_budget) / current_budget.
  • Existing rows with increase_pct already saved still display correctly because we read from proposed_budget first and fall back to the legacy formula only if proposed is null.
  • "Mark No Change" stays as a one-click shortcut that sets Proposed = Current.
  • Notes still warn when |delta| > 10% and notes are empty - that guardrail doesn't change.
  • FA dashboard, Excel export, and totals all read from proposed_budget already, so no downstream changes needed.
