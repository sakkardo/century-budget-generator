Portfolio fix process — visual walkthrough

How the canonical-map fixes work end-to-end, from SharePoint Excel files to dollar values flowing on the dashboard. With real numbers from today's scan.

1) The 4-layer data architecture

The reason fixes don't auto-propagate is that there are four separate layers, and the canonical map is the bridge. When a building's label doesn't match what canonical knows, the dollar values don't flow to the right row.

Layer 1 — Source ledger
Yardi Chart of Accounts
The real general ledger. ~300 distinct GL codes used across the portfolio. Each transaction posts to one GL.
4010-0000 Maintenance 4040-0000 Commercial Rent 5630-0000 Plumbing Repairs 6145-0000 Errors & Omissions Insurance 7025-0000 Flip Tax (expense)
Layer 2 — FA-curated rollup template
SharePoint approved 2026 budget Excel files
One file per building. Contains the row labels each FA used. No GL codes in this file — just human-readable line labels. When the wizard imports a building, these labels become the summary tab's rows.
Maintenance Commercial Rent <-- 148 uses this label Storage Room <-- 140's label for storage income Cable <-- 5 buildings shortened "Cable TV" Operating Assessment <-- 23 buildings; canonical is "Assessment - Operating"
Layer 3 — The bridge (where today's work happens)
GL_TO_SUMMARY_MAP.py — canonical rollup rules
Says "this row label aggregates from these GL prefixes." Has two parts: SUMMARY_ROW_MAP (label → GL prefix list) and LABEL_ALIASES (variant label → canonical label).
"Maintenance": { gl_prefix: ["4010"] } "Commercial": { gl_prefix: ["4040"] } "Storage Income": { gl_prefix: ["4130-0000", "4130-0010"] } "Repairs & Maintenance": { gl_prefix: [54 prefixes] } LABEL_ALIASES = { "Commercial Rent": "Commercial", "Bicycle Storage": "Bicycle Charge", "Operating Assessment": "Assessment - Operating", <-- proposed ... }
Layer 4 — What the FA sees
BudgetSummaryRow records (the summary tab in the dashboard)
One row per (entity, year, label). Has a gl_prefixes_json field that's resolved from the canonical map at import time. The summary tab's dollar values come from summing GL lines that match this prefix list.
Building 148, label="Commercial Rent", gl_prefixes_json=[] <-- empty! No data flows. col3_ytd = $0 After we added LABEL_ALIASES["Commercial Rent"] = "Commercial": gl_prefixes_json=["4040"] <-- prefix populated! col3_ytd = $144,015 <-- dollars flow.

2) What the scanner does

The scanner endpoint at /api/admin/portfolio-label-scan (admin-gated, shipped today) is a read-only probe. It does not change any data — it tells you what would break if you imported each building cleanly.

1

Load active buildings

Reads buildings.csv — 143 buildings (post Monday-filter cleanup yesterday).

2

For each building, find its approved 2026 budget Excel in SharePoint

Calls Microsoft Graph: drives/<drive>/root:/2027 Budget/<entity>:/children. Filters to files matching the "approved" regex pattern. Result: 100 of 143 buildings had a findable file.

3

Download each file (~1-3 MB each)

Streams bytes via Graph API. Saved to a temp file for the parser. Deleted immediately after.

4

Parse with budget_summary_parser.parse_yrlycomp()

Returns a structured list of rows. Each row has a label (and other fields we ignore for this scan — we only want labels).

5

Cross-reference each label against canonical

Checks: is the label in SUMMARY_ROW_MAP? In _CONDO_ROWS? In LABEL_ALIASES? If none of the three, it goes into the unmapped list.

6

Aggregate by frequency

Same unmapped label seen across multiple buildings is grouped. "Operating Assessment" found on 23 buildings is more urgent to fix than a singleton on 1 building.

7

Return JSON gap report

List of unmapped labels with the buildings affected, plus per-building-type stats.

Important: the scanner only reads. Nothing changes. No buildings get touched. It's safe to run as often as you want. Takes ~3 minutes per chunk of 30 buildings (Railway gateway timeout caps it), so we ran 5 chunks today.

3) One fix end-to-end — "Operating Assessment"

The scanner found "Operating Assessment" appearing on 23 buildings. Canonical has the same row labeled "Assessment - Operating". One line of code lifts 23 buildings.

1

Identify in scanner output

Top row of the alias bucket: 23x 'Operating Assessment' → 'Assessment - Operating'

2

Decide: alias vs new canonical row

The label "Assessment - Operating" already exists in SUMMARY_ROW_MAP with gl_prefix: ["4200"]. So we don't need a new row — just an alias from the variant label.

3

Add one line to LABEL_ALIASES

// File: budget_summary/GL_TO_SUMMARY_MAP.py

  LABEL_ALIASES = {
      "Common Charges": "Maintenance",
      "Storage": "Storage Income",
      "Bicycle": "Bicycle Charge",
      "Bike": "Bicycle Charge",
      "Bicycle Storage": "Bicycle Charge",
      "Commercial Rent": "Commercial",
+     "Operating Assessment": "Assessment - Operating",     <-- ADDED (helps 23 buildings)
      "Assessment-Operating": "Assessment - Operating",
      ...
  }
4

Deploy

bash deploy.sh → Railway redeploys → new canonical map live in ~60 seconds.

5

Trigger resolve-aliases on each affected building

POST /api/admin/resolve-summary-aliases/<entity> for each of the 23 entities. The resolver reads each building's BudgetSummaryRow rows, checks each label against canonical + aliases, and re-stamps gl_prefixes_json to match.

6

Result: every "Operating Assessment" row across 23 buildings now has the right GL prefix

The summary aggregation runs at request time — next time anyone opens that building's summary tab, the dollar value is there. No FA action required.

4) How verification works

Three independent mechanisms confirm each fix worked. Any one of them can catch a regression.

1

Re-run the scanner

After deploying a fix, hit /api/admin/portfolio-label-scan again. The fixed label should drop out of the unmapped list. If it doesn't, the alias didn't propagate.

2

Hit the API per building

For any affected building: GET /api/summary/<ec> returns each row's gl_prefixes field. The fixed row should show the canonical prefix list (not []).

3

Visual check — the dashboard

Open the building's summary tab. The fixed row's YTD column shows real dollar values now (was $0). Readiness panel's "orphan GLs" counter drops.

6) What happens next — the full plan with verification at every step

1

I ship the 38 aliases + 5 obvious new canonical rows

One code change. One deploy. Verifiable by re-running the scanner.

2

Run resolve-aliases portfolio-wide

One HTTP call per building → re-stamps gl_prefixes_json on existing rows. Returns count of rows updated per building. Verifiable in the response payload.

3

Re-run the portfolio scanner

The 38 aliased labels should drop from the unmapped list. If they don't, something didn't propagate.

4

Working session resolves the 22 judgment-call labels (Bucket B + D)

Each one: you/FA picks the GL prefix and section. I add to canonical + deploy + resolve-aliases. Same verification chain.

5

I build the per-building override registry

New DB table + admin UI. Lets a per-(entity, label) row override canonical when a building's GL has a unique meaning. Handles the 173 singletons without polluting canonical.

Every step is reversible. Aliases are code — git history. resolve-aliases updates a JSON field; we can rollback by snapshotting before. The summary tab math is recomputed on every request from the (gl_prefixes, GL line) join — no destructive caching. If anything looks wrong, the FA opens the dashboard and immediately sees the wrong dollar value; we revert the canonical change and re-resolve.
