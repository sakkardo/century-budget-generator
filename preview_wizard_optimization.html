4. One backend computes "readiness tier" for both pages

One server-side function categorizes each entity. Wizard + Dashboard both render from it, so they can't disagree. 

// Per-entity readiness tier — server-side computation
//   READY_TO_BUILD = audit confirmed + all 4 Yardi sources ingested,
//                    no wizard_completed_at yet
//   IN_PROGRESS    = some sources ingested but not all,
//                    or audit extracted but not confirmed
//   NEEDS_AUDIT    = files in SP for B/E/Y/A but no audit_2025 PDF
//                    or audit not yet extracted
//   NEEDS_FILES    = some of B/E/Y/A/Au files missing in SP
//   BUILT          = wizard_completed_at IS NOT NULL
//
//   next_action ∈ { "build", "audit_review", "wait", "review_built" }
//   next_step_url = /wizard/<ec>?step=N  with smart step selection

6. Implementation plan — 4 ordered steps

  1. Backend readiness endpoint — extend /api/budgets to include readiness_tier + next_action + next_step_url per entity. Computed from existing data (no new tables). ~30 min
  2. Wizard Select Entity rewrite — replace the Stage column with the tile bar, rewrite filter chips to match readiness tiers, add hero callout + bulk-build button. ~1 hour
  3. Dashboard additions — add the same hero callout + tier filter chips above the existing table. Existing columns, tiles, and Scan button untouched. ~30 min
  4. Bulk-build admin endpoint — POST /api/admin/build-all-ready runs the auto-walk pattern (the same one I demoed manually) across every READY_TO_BUILD entity, streams progress, reports completed/failed. ~45 min

Total: ~2.5-3 hours. None of the existing wizard steps, dashboard tiles, or backend routes change behavior. All additions are layered on top of existing data. Worst-case rollback: revert one or two files, system returns to current state.
